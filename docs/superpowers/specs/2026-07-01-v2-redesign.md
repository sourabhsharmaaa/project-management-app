# V2 Redesign — Design Spec

Date: 2026-07-01

## Overview

Three changes based on evaluation feedback:

1. **Code organization** — introduce service and repository layers
2. **Card ordering** — fractional indexing for performant reordering within a list
3. **Frontend** — React UI to make this a full stack application

---

## 1. Code Organization

### Problem

All logic currently lives in controllers — DB calls, validation, business rules. This makes the code hard to maintain and violates single responsibility.

### Solution

Three-layer architecture per module:

```
Request → Controller → Service → Repository → DB
```

| Layer | Responsibility |
|---|---|
| Controller | Read request, call service, send response |
| Service | Business logic, validation, error throwing |
| Repository | All Prisma/DB calls, nothing else |

### New file structure per module

```
src/modules/<resource>/
  <resource>.routes.js       — URL definitions only
  <resource>.controller.js   — thin request/response handling
  <resource>.service.js      — business logic
  <resource>.repository.js   — all Prisma calls
```

### Error handling

Services throw structured errors:
```js
throw { status: 404, message: 'Card not found' }
```

Controllers catch and respond:
```js
} catch (err) {
  if (err.status) return res.status(err.status).json({ error: err.message })
  res.status(500).json({ error: 'Internal server error' })
}
```

## 2. Card Ordering — Fractional Indexing

### Problem

Cards within a list have no order. Integer positions (1, 2, 3...) are expensive — moving one card requires updating every card's position number (O(n) DB writes).

### Solution

Add a `position` Float field to the `Card` model. Reordering inserts a card between two others by averaging their positions — always one DB write regardless of list size.

### Example

```
Card A → position: 1.0
Card B → position: 2.0
Card C → position: 3.0

Move C between A and B:
  new position = (1.0 + 2.0) / 2 = 1.5

Card A → 1.0
Card C → 1.5  ← only this updated
Card B → 2.0
```

### Schema change

```prisma
model Card {
  position Float @default(0)
  // ... existing fields
}
```

Cards are always returned ordered by `position ASC`.

### New endpoint

```
PUT /cards/:id/reorder
Body: { "afterCardId": 2 }   // move after card 2
      { "afterCardId": null } // move to beginning of list
```

**Logic:**
- `afterCardId: null` → position = first card's position / 2
- `afterCardId` provided → position = (afterCard.position + nextCard.position) / 2, or afterCard.position + 1 if last

### Existing endpoint stays

`PUT /cards/:id/move` remains unchanged — it moves a card between lists (changes `boardListId`).

## 3. Frontend — React

### Tech

- **Vite + React** — fast dev setup
- **Plain CSS** — no UI library
- Lives in `client/` at project root
- Runs on port 5173, proxies API calls to backend at port 3000

### Views

| View | Route | Description |
|---|---|---|
| Home | `/` | All boards, create board |
| Board | `/boards/:id` | Lists and cards, create list/card |

### Component structure

```
client/src/
  pages/
    Home.jsx          — board list + create board form
    Board.jsx         — lists + cards view
  components/
    BoardCard.jsx     — single board preview
    List.jsx          — list column with cards
    Card.jsx          — card with assign/move/reorder actions
  api/
    index.js          — all fetch calls in one place
  App.jsx
  main.jsx
```

### API layer

All backend calls go through `client/src/api/index.js`:

```js
const BASE = 'http://localhost:3000'

export const getBoards = () => fetch(`${BASE}/boards`).then(r => r.json())
export const createBoard = (data) => fetch(`${BASE}/boards`, { method: 'POST', ... })
// etc.
```

No fetch calls scattered across components — one place to change if the base URL changes.

### Features

- View all boards
- Create a board
- View a board with all lists and cards
- Create a list inside a board
- Create a card inside a list
- Move a card to another list
- Reorder a card within its list
- Assign/unassign a user to a card

### What's excluded

- Drag and drop — move/reorder done via buttons/dropdowns
- Authentication
- Delete UI (API supports it, frontend keeps it simple)
